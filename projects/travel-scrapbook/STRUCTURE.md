## What This App Does

Travel Scrapbook is for people who research trips by collecting links — a Reddit thread about ramen shops, an Instagram reel about a hidden bar, a TripAdvisor page found on the couch. Capture is everywhere the user already is: **share from any phone app** (Android share sheet via installed PWA, iPhone via a "Scrap it" Shortcut), a bookmarklet, or an in-app paste box — **every place enters via a URL** (or the community pool); there is no manual entry. The backend scrapes the page, asks Gemini to extract **every place it mentions** (one reel can fan out into several), dedupes each into a canonical **place** (the source of truth — name, category, coordinates, Google Maps link) with the URLs attached as **sources** ("how you stumbled on it"), and geocodes with Nominatim (OpenStreetMap, free). New finds land **staged on a nearby trip** (a "Needs review" section the user approves) or in the **inbox** when no trip matches. Inside a trip, scraps appear as sticker-style cards ("plans"); the user can add anchors with dates (arrival/departure, hotel stays), flip to a **day-by-day timeline** that slots plans between the markers ("near your Day 2 hotel"), **sort a checkpoint-aware route** that clusters plans around each stay/travel checkpoint (nearest-neighbor + 2-opt over haversine distances) and drops the currently-unscheduled ones onto that checkpoint's timeline days, open the route as multi-stop Google Maps directions links, or download a CSV that imports into Google My Maps. A **community pool** shares canonical places (never notes/ratings/identity) across users, and a 7-step tour auto-launches on first login.

Every saved place carries the owner's own **rating** — Booked / Must do / Interested / Could skip — settable from the Wander List or any trip (`scraps.rating`). Trips can be **shared** with other travelers as a viewer or collaborator (invite → accept); on a shared trip every member sets their own **Vibe** on each place (same four levels) which rolls up into a group consensus, and each place shows who added it. The owner's rating doubles as their vibe: setting a rating on an in-trip place upserts the owner's vibe row server-side, so consensus includes them without a second control.

**Core objects:** Trip, Scrap (a saved place in a trip or the inbox; called a "plan"/"todo" inside a trip), **Place** (canonical, deduped; also the community-pool unit), **Source** (capture event), Anchor — surfaced in the UI as a **Checkpoint**: a **Stay** (lodging with check-in/out) or **Travel** (arrival/departure endpoints, or any number of mid-trip legs for multi-city), doubling as route endpoint + dated **timeline marker** — Category, **TripMember** (a viewer/collaborator on a shared trip), **Vibe** (a traveler's per-place take → group consensus; the owner's take is the scrap's **rating**).

## Data Model

All tables RLS-enabled, granted to `travelscrapbook_role`; backend-only access via service role (no Data API grants). The hot reads are backed by **single-round-trip Postgres RPCs** (migration `015`, inventory in `db/functions/travelscrapbook.sql`): `travelscrapbook_trip_bundle` (the whole trip screen), `_trips_list`, `_inbox_bundle`, `_visited_page`, `_community_places`, `_scrap_card` (cheap mutation echo), `_set_route_positions` (bulk write) — each takes the viewer's UUID and enforces owner/member access internally (service role bypasses RLS); all are revoked from `anon`/`authenticated`. The **place is the source of truth**; sources record how the user found it; a scrap is "the user's saved place, in a trip or the inbox".

- **travelscrapbook_profiles** — Supabase Auth profile. `id uuid PK → auth.users ON DELETE CASCADE`, `display_name text`, `username text UNIQUE`, `is_admin bool`, `tutorial_seen_at timestamptz` (NULL = the web app auto-launches the tour once, `010`), `created_at`.
- **travelscrapbook_categories** — seeded option set. `slug text PK` (restaurant, cafe, bar, sight, activity, shop, lodging, hike, other, + checkpoint categories airport/train_station/car_rental/transport), `label text`, `icon text` (sprite slug → `assets/sprites/categories/travel-scrapbook-cat-<icon>.svg`), `sort_order int`. `hike` (021) is the category for trail imports — links from Komoot / AllTrails / Strava / etc. are recognized (`services/trails.py`) and forced to `hike`, pinned at the trailhead (no track geometry; trailhead point only).
- **travelscrapbook_trips** — `id uuid PK`, `user_id → profiles CASCADE`, `name text`, `destination text`, `cover_icon text DEFAULT 'plane'` (sprite slug), **geographic scope** `scope_level text CHECK ('region','country','city') DEFAULT 'city'` + `dest_city/dest_region/dest_country/dest_country_code` (match values derived from the destination geocode; `dest_region` = the destination country's macro-region, so a region-scoped trip spans a whole country grouping), `start_date date`, `end_date date`, `notes text`, **geocoded destination** `lat/lng`, `geocode_confidence`, `geocode_display_name`, `destination_geocoded_at` (NULL = never attempted; drives lazy backfill), timestamps. Index `(user_id)`. Scope drives tag-based staging + the trip candidates panel; city scope stays distance-based.
- **Checkpoints are unified into the place/scrap model (`020`)** — a checkpoint is a **place** (identity, deduped, community-poolable) + a **scrap** (the user's saved, visitable copy) + a **role-bearing `scrap_trips` membership** (`role CHECK ('start','end','stay','travel')`; NULL = ordinary plan). The transport mode (old `anchor.type`) is the place's **category** (`airport`/`train_station`/`car_rental`/`transport`; stay → `lodging`); dates ride the membership (`plan_date`/`plan_time`; stays add `plan_end_date` = check-out). One start + one end per trip via partial unique `(trip_id, role) WHERE role IN ('start','end')`; `same_as_start` reuses the start's scrap for the end membership (its own date). The `/anchors` API keeps its paths + `AnchorResponse` shape — the anchor id is the **membership id**, and responses/trip-bundle `anchors[]` are **synthesized** (SQL in `travelscrapbook_trip_bundle`; Python twin `services/checkpoints.synthesize_anchor` — keep in step). Checkpoint scraps surface on the Wander List / Visited / Community under **"Stays & transport"** (rule: `categories.is_checkpoint`), are excluded from plan candidates/wishlist pickers and the nav badge, and past-trip checkpoints were backfilled as visited. **`travelscrapbook_anchors` is FROZEN** (backfilled by `020`, no reads/writes; dropped by `021` after soak).
- **travelscrapbook_sources** — one capture event. `id uuid PK`, `user_id → profiles CASCADE`, `url text`, `url_normalized text` (tracking params/fragment stripped; UNIQUE per user), `source_domain`, `status CHECK ('processing','ready','failed')`, `error_kind` (network/blocked/llm/no_place), `captured_via CHECK ('paste','bookmarklet','share','shortcut')`, `shared_text` (share-sheet caption — extra LLM context, key for blocked Instagram pages), `capture_notes` (user note at capture, copied onto created scraps), `trip_hint_id → trips SET NULL`, `og_title/og_description/og_image_url`, timestamps.
- **travelscrapbook_regions** — reference data: `country_code text PK` (ISO-3166 alpha-2, lowercase) → `region text` (UN M49 subregion, e.g. Greece→Southern Europe, Japan→Eastern Asia). Seeded in `006`; read backend-only to tag places/trips with a macro-region. Cached in-process (`ts.regions`).
- **travelscrapbook_places** — canonical place, per-user (osm identity recorded for future global/cross-user dedupe). `id uuid PK`, `user_id → profiles CASCADE`, `name text`, `name_normalized text` (accent/case/punct-folded dedupe key), `city/country`, `country_code` (ISO alpha-2), `region` (**macro-region** = the country's UN subregion, a grouping of countries — derived via `travelscrapbook_regions`, not admin-1), `category → categories DEFAULT 'other'`, `lat/lng`, `geocode_confidence`, `geocode_display_name`, `osm_type/osm_id`, `maps_url`, timestamps. Index `(user_id, name_normalized)`. Names come back in English (`accept-language=en`).
- **travelscrapbook_place_sources** — N sources ↔ N places join (`place_id`, `source_id`, PK both, CASCADE both).
- **travelscrapbook_capture_tokens** — iOS Shortcut auth. `id uuid PK`, `user_id → profiles CASCADE`, `token_hash text UNIQUE` (sha256 hex — deterministic so /capture can look up BY token; safe for 256-bit random tokens), `created_at`, `last_used_at`, `revoked_at` (soft revoke; one active per user, enforced app-side).
- **travelscrapbook_scraps** — the user's saved place. `id uuid PK`, `trip_id → trips CASCADE` **nullable** (NULL = wishlist/inbox), `user_id → profiles CASCADE`, `place_id → places CASCADE`, `status CHECK ('inbox','staged','approved')` (staged = auto-matched to a trip, awaiting review; only approved scraps route/export), `notes`, `rating text CHECK (NULL,'booked','must_do','interested','could_skip')` (the owner's own priority, settable anywhere — replaced `is_favorite` in `008`; backfilled from the owner's vibe, then hearts→must_do), `visited_at timestamptz` (NULL = on the wishlist; set = visited → surfaces in the Visited view, excluded from the wishlist + nav badge), `skipped_at timestamptz` (`016`: **timeline-only** "Skipped" outcome — greys/tags the plan on the Timeline but, unlike `visited_at`, does NOT leave the wishlist or Visited view; mutually exclusive with `visited_at`), `route_position`, `plan_date date` + `plan_time time` (timeline slot, trip scraps only; cleared on unassign, `009`), timestamps. Indexes `(trip_id)`, `(user_id)`, `(user_id, status)`, `(place_id)`, `(user_id, visited_at)`, `(trip_id, plan_date)`.
- **travelscrapbook_trip_members** — trip sharing. `id uuid PK`, `trip_id → trips CASCADE`, `user_id → profiles CASCADE`, `role text CHECK ('viewer','collaborator')` (viewer = read + vibe; collaborator = read + vibe + add places/edit the shared route), `status text CHECK ('pending','accepted','declined') DEFAULT 'pending'` (only `accepted` grants access — the invite → accept flow), `invited_by → profiles SET NULL`, `created_at`, `responded_at`. `UNIQUE (trip_id, user_id)`; indexes `(user_id, status)`, `(trip_id)`. **The owner is NOT a member row** — ownership stays on `trips.user_id`; access = owner ∪ accepted members.
- **travelscrapbook_scrap_vibes** — each traveler's **Vibe** on a place (the group-consensus input). `id uuid PK`, `scrap_id → scraps CASCADE`, `user_id → profiles CASCADE`, `level text CHECK ('booked','must_do','interested','could_skip')`, timestamps. `UNIQUE (scrap_id, user_id)` (one vibe per person per scrap); index `(scrap_id)`. Present on all trips (a solo trip just has one voter). Because scraps are per-user, two collaborators saving the same place get two cards, each with its own vibes/consensus.
- **travelscrapbook_scrap_trip_dismissals** (`018`) — "the user already resolved this suggestion for this trip". `id uuid PK`, `scrap_id → scraps CASCADE`, `trip_id → trips CASCADE`, `created_at`, `UNIQUE (scrap_id, trip_id)`; index `(trip_id)`. Written on **every** membership removal (`unassign_scrap`, `set_scrap_trips` drop-branch) so a removed place doesn't re-appear in the "Suggested plans" panel — the marker survives the membership deletion that a `scrap_trips` row can't. Excluded by the candidates query in `travelscrapbook_trip_bundle` + `list_trip_candidates`. The manual "Add from your Wander List" picker is unfiltered, so a dismissed place can always be re-added by hand.

## Trip sharing, roles & vibes

- **Share a trip** with another user by username as a **viewer** or **collaborator** (invite → accept). Invites are pending until the invitee accepts; only accepted members see the trip. Non-destructive removal/leave — a departing member's saved places stay on the trip (still shown "added by X").
- **Roles.** Owner: everything, incl. edit trip settings, manage members, delete the trip. Collaborator: read + add places (capture/assign onto the shared trip) + edit the shared route (anchors, optimize). Viewer: read + set their own vibe only. Enforced in `access.py get_accessible_trip(...)` (owner-or-member gate; `need_write`/`need_owner` flags) — the single chokepoint replacing the old owner-only `get_owned_trip`. Scrap *content* mutations (edit/delete a place) stay own-scrap-only (`get_owned_scrap`) so no one edits another traveler's places.
- **Vibes → consensus.** Every member (viewers included) sets their own vibe on each place; the card shows a per-member chip row + a consensus headline (top count, tie-break booked > must_do > interested > could_skip). "Added by X" shows who saved each place on a shared trip. Vibes + consensus are attached in `hydrate_scraps(..., with_vibes=True)` on the trip surfaces (2 extra batched round-trips, no N+1).

## API Endpoints

All under `/api/v1/travel_scrapbook`, Supabase bearer auth (profile auto-created) except health; `POST /capture` also accepts a personal capture token (`tsc_…` prefix).

- `GET /health` — health check, no auth
- `GET /me` — profile bootstrap (incl. `tutorial_seen`) + category list; `PATCH /me` — update display_name; `POST /me/tutorial-seen` — never auto-launch the tour again (idempotent)
- `GET /trips` — owned **plus accepted-shared** trips with scrap counts, each tagged with the caller's `role` + `owner_display_name` (lazily backfills destination geocodes + `dest_*` components, inferring legacy scope); `POST /trips` (accepts `scope_level`; geocodes destination synchronously, inferring scope from the destination when none given); `GET /trips/{id}` — the whole trip screen in ONE DB round-trip (`travelscrapbook_trip_bundle`): trip + anchors + `scraps` (approved) + `staged_scraps` + `members` (roster incl. pending) + `candidates` (the viewer's scope-matched wishlist adds; write roles only), plus the caller's `role`/owner (readable by owner or member; vibes hydrated in); `PATCH /trips/{id}` (owner only; re-geocodes a changed destination, honors `scope_level`); `DELETE /trips/{id}` (owner only)
- `POST /trips/{id}/anchors` — create a checkpoint + geocode synchronously (accepts `type` + `anchor_date`/`anchor_time` for start/end/travel, `stay_date`/`stay_end_date` for stay, an optional `maps_url` — a Google Maps link parsed to itself for coords + city/region/country, no AI — or `same_as_start` on an end anchor to copy the start's place + type — not its date — without geocoding); dates validated against the trip's bounds (lenient when undated); `PATCH /anchors/{id}` — edit (re-resolves on a changed `query` or a pasted `maps_url`, re-validates dates); `DELETE /anchors/{id}`
- `POST /capture` → 202 — the single silent-capture entry: `{url?, text?, title?, trip_id?, via?, notes?}`; URL taken from `url` or extracted from `text` (Android share sheets put it there); dedupes on `(user, url_normalized)` (re-capture reuses the source; failed/stale resets and re-runs); processing via BackgroundTasks
- `POST /capture-token` → 201 (plaintext shown once; replaces prior token); `GET /capture-token` — status; `DELETE /capture-token` — revoke
- `GET /inbox?region=&country=&city=&limit=24&offset=0` — the wishlist in ONE DB round-trip (`travelscrapbook_inbox_bundle`: SQL-side filter/facets/pagination + sources + the geocoded trips that feed suggestions): `{processing_sources, failed_sources, scraps, total, facets, inbox_count}` — one filtered PAGE of unvisited inbox scraps (each carries `suggestions`: nearest ≤3 trips within 200 km, plus country/region trips by tag match — computed against ONE preloaded trips list, not per scrap) plus `facets` (all three levels always populated: `regions`; `countries` narrowed to the selected region when given; `cities` narrowed to the selected country/region when given) and `inbox_count` (the global nav-badge count, so the web app skips the separate count call); sweeps sources stuck processing >10 min → failed (one batched UPDATE); `GET /inbox/count` — nav badge (unvisited only; still used at boot)
- `GET /visited?region=&country=&city=&limit=&offset=` — one filtered page of scraps with `visited_at` set (any trip or the wishlist), most-recently-visited first, + `total` + `facets` — the Visited view
- `POST /sources/{id}/retry` → 202; `DELETE /sources/{id}` — dismiss; `GET /sources/{id}/scraps` — a capture's live `{status, error_kind, scraps}` (drives the "watch it import" cards on the share success screen)
- `GET /scraps/{id}`; `GET /trips/{id}/scraps` — hydrated with place fields (incl. `place_region`) + source chips
- `GET /trips/{id}/candidates` — unvisited wishlist scraps whose location matches the trip's scope (city/country/region), for the inline "Suggested plans" panel; same predicate as auto-staging; excludes pairs the user already resolved (`travelscrapbook_scrap_trip_dismissals`, `018`)
- `GET /trips/{id}/wishlist` — ALL unvisited wishlist scraps + a `fits_scope` flag (matches sort first), for the trip's "Add plans → From your Wander List" picker (not scope-filtered — add anything); `POST /trips/{id}/assign-scraps` `{scrap_ids}` — bulk-add them as approved. **In a trip, scraps are called "plans" in the UI** (internal identifiers unchanged). There is **no manual place entry** — every place enters via a URL capture (removed `POST /trips/{id}/plans` in favor of URL-only intake).
- `PATCH /scraps/{id}` — name/category edits write to the canonical **place** row; notes/`visited`/`skipped`/`plan_date`/`plan_time` stay on the scrap (`visited` → `visited_at` now()/NULL, `skipped` → `skipped_at` now()/NULL; the two are mutually exclusive — setting one clears the other; scheduling requires a trip → 400 otherwise, day validated against trip bounds, explicit null clears). **City/country/region are read-only** — they derive from the place's pin, so the only way to move a place is to pass a new `maps_url` (city/region/country re-derive from the Maps link; no manual location typing, no `regeocode` flag)
- **Rating** — `PUT /scraps/{id}/rating` `{level}` / `DELETE /scraps/{id}/rating` — set/clear the owner's own priority (owner-only). When the scrap is in a trip, the rating also upserts/deletes the owner's vibe row (one-way sync: rating→vibe), so group consensus includes the owner without a second control.
- `GET /trips/{id}/timeline` — day-by-day timeline (any member): `{days: [{date, day_number, markers, plans}], unscheduled, reason?}`. Markers = dated anchors flattened to arrival / check-in / check-out / departure events; plans sort by time within the day (untimed after timed, markers first on ties); unscheduled unvisited plans carry a `suggestion` (nearest located marker within 25 km → "near your Day N stay"). `reason: no_dates` when nothing is dated. **The web app no longer calls this** — it computes the identical payload locally from the trip bundle (`web/domain/timeline.js`, a port of `services/timeline.py`; keep the two in step) so tab switches and schedule changes are zero-network. The endpoint stays for other clients.
- **Community pool** — `GET /community/places?q=&region=&country=&city=&category=&limit=24&offset=0` — one filtered page of places any traveler has scrapped, aggregated across users (geocoded only, grouped by OSM identity else name+country, most-saved first) + `total` + geo drill-down `facets`: canonical facts + `saved_by_count`/`source_count`/`sample_sources`/cover image — **never notes, ratings, vibes, or user identity**. `POST /community/places/{place_id}/save` `{trip_id?}` → 201 — copies the canonical fields into the caller's own place (no re-geocode; sources are NOT copied) and scraps it onto the trip (approved) or the Wander List (inbox); reuses an existing place/scrap.
- **Trip membership (multi-trip, `013`)** — `POST /scraps/{id}/assign` `{trip_id}` → add one approved membership (additive); `PUT /scraps/{id}/trips` `{trip_ids}` → reconcile the exact set (multi-select picker); `POST /scraps/{id}/trips/{trip_id}/approve` (staged→approved, else 409); `DELETE /scraps/{id}/trips/{trip_id}` → remove from that one trip (its route/schedule/vibes cascade; place stays on the Wander List + other trips); `PATCH /scraps/{id}/trips/{trip_id}/schedule` `{plan_date?,plan_time?}` → per-trip timeline slot; `POST /trips/{id}/approve-all` — approve every membership the caller staged; `DELETE /scraps/{id}` — delete the place entirely (all memberships cascade)
- `POST /trips/{id}/route/optimize` — `{scrap_ids?, priority_only?, include_visited?}` (priority_only = booked/must-do plans only; visited plans skipped unless include_visited); NN + 2-opt with start/end anchors; **approved scraps only**; persists `route_position`; returns ordered scraps + leg/total km + skipped (ungeocoded)
- `GET /trips/{id}/export/maps-links?include_visited=&date=` — JSON `{legs: [{label, url, stop_count}]}` of `google.com/maps/dir/...` URLs (≤10 stops/leg, legs overlap at endpoints); visited plans skipped by default
- `GET /trips/{id}/export/csv?include_visited=&date=` — text/csv attachment (name, category, address, lat, lng, notes, url) for Google My Maps import; visited plans skipped by default
- `GET /trips/{id}/export/markdown?include_visited=&date=` — text/markdown attachment: a readable itinerary (trip header + start/end anchors + each place's category, address, notes, Google Maps link); visited plans skipped by default
- `GET /trips/{id}/export/kml?include_visited=&date=` — `application/vnd.google-earth.kml+xml` attachment: one named `<Placemark>` per **geocoded** place (imports into Google My Maps / Google Earth); visited plans skipped by default. Note: Google Maps *Saved Lists* have no file import — KML targets My Maps / Earth
- All four exports accept an optional **`date=YYYY-MM-DD`** filter → only that day's scheduled plans (`plan_date`), time-ordered; markdown/KML are headed by the date, filenames suffixed ` - <date>`. Undated "Anytime" plans appear only in the whole-trip export (no `date`).
- **Sharing** — `GET /trips/{id}/members` (owner + members, incl. pending); `POST /trips/{id}/members` `{username, role}` → 201 (owner; invite by username, pending); `PATCH /trips/{id}/members/{userId}` `{role}` (owner); `DELETE /trips/{id}/members/{userId}` (owner removes, or member leaves); `GET /invitations` — my pending invites; `POST /trips/{id}/invitation/respond` `{action: accept|decline}`
- **Vibes (per trip, `013`)** — `PUT /scraps/{id}/trips/{trip_id}/vibe` `{level}` — set/replace my vibe on that place for that trip (any member incl. viewer); `DELETE /scraps/{id}/trips/{trip_id}/vibe` — clear mine; both return the re-hydrated scrap (in the trip's context) with the updated consensus. The owner's `rating` still doubles as their vibe — set/clear syncs the owner's vibe row across **every** trip the place is in.

## Key Business Logic

- **Capture pipeline** (`services/enrichment.py::process_source`, background task, never raises): fetch page (browser UA; on 403/login-wall degrade to URL-slug + share-sheet-text context — common for Instagram/Reddit) → Gemini extracts an **array** of places (max 8) plus an optional **booking classification** as strict JSON `{places: [{place_name, city, country, category, geocode_query, confident}], booking: null | {kind: stay|travel, label, location, start_date, end_date, time, transport_type}}` → per place: Nominatim fallback chain `name,city,country` (high) → `name,country` (medium) → `city,country` (low, centroid) → none → `find_or_create_place` dedupe → link `place_sources` → create a scrap **only if the user has no scrap for that place yet**. Zero places → source `failed/no_place`. Source ends `ready`/`failed`.
- **Booking links → checkpoints** (`_materialize_checkpoint`): a URL Gemini classifies as a lodging/transport booking (hotel/Airbnb reservation or listing, flight/train/ferry/car itinerary) captured **with a trip context** (quick-paste on the trip, share with trip hint) becomes a **checkpoint** on that trip — role stay/travel, label + geocoded location + dates/time filled in from the page — instead of place scraps (re-capturing the same booking updates the existing checkpoint by label). Without a trip there's nowhere to hang it, so it falls through to the normal place flow (the LLM lists the lodging/destination in `places` too). The trip poll counts anchors as well as scraps, so a landed checkpoint stops it.
- **Place dedupe** (`services/places.py`): same `name_normalized` (NFKD accent-fold, lowercase, punctuation-stripped, leading "the" dropped) AND (both geocoded within 0.5 km, or coords missing with city agreeing/absent). Never coordinate-only. On merge, NULL fields fill in from the new extraction. Per-user scope; `osm_type/osm_id` is the forward path to global cross-user dedupe.
- **Trip auto-staging**: trip destinations geocode on create/update (lazy backfill on list). A confident, geocoded place within **100 km** of a destination staged onto the nearest matching trip (upcoming/undated trips beat past ones); explicit `trip_id` at capture → approved directly; otherwise inbox. Inbox cards suggest the nearest ≤3 trips within 200 km.
- **URL dedupe**: `normalize_url` (host lowercased, www/fragment/trailing-slash stripped, `utm_*`/`fbclid`/`igsh`/`si`/`gclid` params dropped, query sorted); UNIQUE `(user_id, url_normalized)` — re-sharing the same reel attaches to the existing source instead of duplicating.
- **Capture auth**: `get_capture_user` routes on the `tsc_` prefix — capture-token lookup (sha256) for iOS Shortcut traffic, Supabase JWT otherwise. Tokens: `tsc_` + `secrets.token_urlsafe(32)`, shown once, one active per user, soft-revoked.
- **Nominatim courtesy**: module-level asyncio lock + monotonic timestamp enforce ≥1.1s between calls; results cached 30 days (cache.py ns `ts.geocode`); descriptive User-Agent. Never add a Google geocoding key — links are plain URLs.
- **Community aggregation** (`travelscrapbook_community_places` RPC, migration `015`): grouping (OSM identity when Nominatim supplied one else `(name_normalized, country_code)`), most-complete representative pick, distinct-saver counts, filter/facets/pagination, and page-only source chips all run in SQL — one round trip (replaced the 2000-row fetch + Python aggregation in `services/community.py`, which now keeps only the write-side `copy_place_for_user`). Privacy invariant: places are shared as facts about the world; scraps/notes/ratings/vibes/identity never leave their owner. Saving copies canonical fields into the caller's own place row (reuse by OSM id else name, no Nominatim call).
- **Timeline** (`services/timeline.py::build_timeline`, pure math over hydrated rows — no DB/geocode calls): day range = trip dates (fallback: min/max dated marker/plan); markers flatten from dated anchors (a stay contributes check-in + check-out); within a day items sort `(time is None, time)` with markers before plans on ties. **Slot suggestions**: each unscheduled, geocoded, unvisited plan gets the nearest located marker within `TIMELINE_SUGGEST_RADIUS_KM = 25` as "Day N · near X". A plan rated `booked` with a `plan_time` renders as a concrete booking (client-side rule). **Timeline layout** (`ui/trip-timeline.js`): bookended by the trip endpoints — ghost "+ Arrival"/"+ Departure" buttons until each anchor exists, then the anchor card with an edit pencil (anchor-editor edit mode → `PATCH /anchors/{id}`); arrival/departure render as bookends; stay check-in/out and mid-trip **travel** markers appear inside day cards, each with an edit pencil; unscheduled plans **auto-place into their suggested day** as dashed rows (pin = one-tap `plan_date`); plans with no suggestion collect in "Anytime"; a "+ Checkpoint" affordance (stay or travel) sits mid-timeline. The layout renders even on an undated trip so the bookends stay actionable. **Per-day plans sort by ROUTE ORDER** (`route_position` from "Sort my route" — the best point-to-point driving sequence, not priority; unrouted plans fall to the end, then timed-before-untimed / time / name; finished plans sink to the bottom); stay/travel markers render first, chronologically. **Plan-row gestures** (`widgets/timeline-gestures.js`, one Pointer-Events state machine per row): a leading **checkbox** cycles the outcome — clear → **Visited** → **Skipped** → clear — greying + tagging the place (`ScrapDomain.setTimelineOutcome`, optimistic); **swipe right** opens the day picker (schedule); **swipe left** unschedules a scheduled plan (clears `plan_date` — "off timeline", stays in the trip); **press-and-hold** picks the card up to drop on any day card (`data-day-date` drop targets). Buttons inside a row (checkbox, pin) opt out of the gesture so their tap still fires.
- **Route optimizer** (`services/optimizer.py`): haversine matrix; nearest-neighbor seeded at start anchor (or first scrap), 2-opt improvement with fixed endpoints; open path when no end anchor. Approved scraps only; ungeocoded scraps are skipped and reported.
- **Maps export chunking**: Google directions URLs cap ~10 waypoints; legs overlap (last stop of leg N = first stop of leg N+1).
- **File exports** (`services/exports.py`): `build_csv` (My Maps columns), `build_markdown` (itinerary text; per-place Google Maps link = `maps_url` else a `.../maps/search/?api=1&query=lat,lng` fallback), `build_kml` (one `<Placemark>` per geocoded place, CDATA description). Routes in `export_routes.py` share `_ordered_scraps` (route_position → created_at) + `_safe_filename`. Frontend: `ExportDomain` (`domain/export.js`) owns the blob-download mechanics; `ExportMenu` (`widgets/export-menu.js`) is the toolbar Download popup.
- **Bookmarklet = popup window**, not in-page fetch: third-party CSP (Instagram/Reddit) blocks XHR from injected JS, and the popup is our origin so the Supabase session just works. No tokens ever touch third-party pages. `/share` follows the same pattern (localStorage stash across the OAuth hop).
- **Hydration** (`services/hydrate.py`): scraps serve flat place fields + source chips in 3 DB round-trips regardless of count (scraps → places → place_sources+sources); `og_image_url` = first non-null image among the place's sources.
- **Stuck processing**: BackgroundTasks are lost on worker restart; `GET /inbox` sweeps sources processing >10 min to `failed/network`; trip/inbox polls time out at 45 s.
